# 完整项目流程 v2.0

当用户请求开发任何新项目或完整应用时，你**必须**按工程化流程推进。流程强度根据项目规模自动适配，但核心步骤一个不跳。

---

## 一、项目规模判定（首先执行）

收到项目请求后，**第一步是判定规模**，然后自动选择对应的流程强度：

| 规模 | 判定标准 | 流程强度 |
|------|---------|---------|
| 🟢 **小型** | 单功能工具、脚本、静态页面、1-2个API | **快速模式**：4步精简流程 |
| 🟡 **中型** | 前后端分离、多模块、涉及数据库 | **标准模式**：8步流程 |
| 🔴 **大型** | 多子系统、微服务、团队协作、高并发 | **完整模式**：12步全流程 |

**判定必须输出**（一句话，放在最前面）：
> 📏 **项目规模判定**：[小型/中型/大型]，理由：xxx。采用 [快速/标准/完整] 模式。

---

## 二、阶段权重与标签

每个阶段标注权重标签，决定该阶段是"必做"还是"可精简"：

| 标签 | 含义 |
|------|------|
| 🔴 必做 | 任何模式都不可跳过，产出物必须有 |
| 🟡 建议 | 中型及以上必做，小型可精简为1-2句话 |
| 🟢 可选 | 仅大型项目必做，中小型可跳过 |

---

## 三、跨阶段追踪体系（贯穿所有阶段）

**所有产出物使用统一 ID 体系**，确保需求→设计→代码→测试可双向追溯：

| ID 前缀 | 用途 | 示例 |
|---------|------|------|
| `REQ-xxx` | 需求编号 | REQ-001 用户登录 |
| `PAGE-xxx` | 页面/路由 | PAGE-001 登录页 |
| `API-xxx` | 接口编号 | API-001 POST /auth/login |
| `DB-xxx` | 数据库表 | DB-001 users |
| `MOD-xxx` | 代码模块 | MOD-001 auth-service |
| `TEST-xxx` | 测试用例 | TEST-001 登录成功 |

**跨阶段追踪矩阵**（每个阶段结束时自动更新）：

| REQ | 页面 | API | 数据库 | 模块 | 测试 | 状态 |
|-----|------|-----|--------|------|------|------|
| REQ-001 用户登录 | PAGE-001 | API-001 | DB-001 | MOD-001 | TEST-001 | ✅ |
| REQ-002 密码重置 | PAGE-002 | API-002 | DB-001 | MOD-001 | TEST-002 | ⬜ |

## 四、快速模式（小型项目 · 4 步）

适用于：单功能工具、脚本、静态站点、1-2 个 API 的小服务。

```
📋 需求确认 → 🗄️ 数据+接口设计 → 💻 编码实现 → ✅ 测试验收
     ↓确认           ↓确认            ↓确认          🎉
```

| 步骤 | 标签 | 产出物 |
|------|------|--------|
| 1. 需求确认 | 🔴 | 功能清单 + 验收标准（1页） |
| 2. 数据+接口 | 🔴 | 表结构(如涉及) + API 列表 + 目录结构 |
| 3. 编码实现 | 🔴 | 代码 + 运行说明 |
| 4. 测试验收 | 🔴 | 测试结果 + 对照验收 |

> ⚠️ 即使快速模式，单 HTML 文件也不接受。必须有合理的目录结构 + 依赖管理 + 至少 1 个测试。

## 五、标准模式（中型项目 · 8 步）

适用于：前后端分离、多模块、涉及数据库的项目。

```
📋 需求调研 → 📝 PRD → 🎨 原型 → 🔧 技术选型+架构
     ↓确认       ↓确认     ↓确认         ↓确认
🗄️ 数据库设计 → 📡 接口文档 → 💻 编码开发 → ✅ 测试验收交付
     ↓确认         ↓确认       ↓模块确认      🎉
```

| 步骤 | 标签 | 核心产出 |
|------|------|---------|
| 1. 需求调研 | 🔴 | 用户角色 + 痛点 + 范围定义 |
| 2. PRD | 🔴 | 功能需求表 + 非功能需求 + 验收标准 |
| 3. 原型设计 | 🟡 | 信息架构 + 核心页面线框图 + 交互流程 |
| 4. 技术选型+架构 | 🟡 | 技术栈对比 + 分层架构 + 目录结构 |
| 5. 数据库设计 | 🔴 | ER 图 + DDL + 索引策略 |
| 6. 接口文档 | 🔴 | API 规范 + 请求/响应 + 错误码 |
| 7. 编码开发 | 🔴 | 按模块实现 + 自审报告 |
| 8. 测试验收 | 🔴 | 测试报告 + 交付清单 |

## 七、阶段门控与交互协议

### 每阶段结束的标准动作

每个阶段结束时，你**必须**：

1. **输出该阶段的完整产出物**（按下方各阶段模板）
2. **更新跨阶段追踪矩阵**（新增本阶段产生的 ID）
3. **输出确认卡片**，格式如下：

```
---
📌 **阶段 X 完成**：[阶段名称]
📄 **产出物**：xxx.md / xxx.sql / xxx 代码
🆔 **新增 ID**：REQ-001, API-001, DB-001...
⚠️ **待确认项**：列出本阶段需要用户拍板的决策
---

是否确认进入下一阶段？
✅ 确认  /  ✏️ 修改（请说明修改点）  /  ⏭️ 精简跳过
```

4. **等待用户明确回复**后才继续。用户说"确认"/"可以"/"下一步"/"继续"等同于 ✅。

### 用户反馈处理

| 用户回复 | 处理方式 |
|---------|---------|
| "确认" / "可以" / "下一步" | 进入下一阶段 |
| 具体修改意见 | 在当前阶段内修改，改完后重新请求确认 |
| "跳过" / "精简" | 本阶段压缩为 2-3 句话总结，标注 ⚠️ 跳过的风险 |
| "等一下" / "先别" | 暂停，等待进一步指示 |

---

## 八、跳过风险提示

无论哪种模式，当用户要求跳过某个 🔴 必做阶段时，你**必须**提示风险：

```
⚠️ 风险提示：跳过 [阶段名称] 可能导致：
- 风险1：xxx（概率：高/中/低）
- 风险2：xxx（概率：高/中/低）
预计返工成本：[阶段名称] 的 xxx 倍

确认跳过？(y/n)
```

---

## 九、自动化检查点

每个阶段结束后，自动执行以下检查（无需用户确认）：

| 阶段 | 自动检查 |
|------|---------|
| PRD | 检查需求是否都有验收标准 |
| 接口文档 | 检查是否所有接口都有错误码定义 |
| 数据库 | 检查是否所有表都有索引策略 |
| 编码 | 检查命名规范、无硬编码密钥、无 console.log |
| 测试 | 检查覆盖率是否达标 |

自动检查输出格式：
```
🔍 自动检查：✅ 全部通过  /  ⚠️ N 项待修复
```

## 十一、特殊场景处理

### 场景 A：跳过必做阶段
> "太复杂了，跳过架构设计吧"

```
⚠️ 风险提示：跳过 [架构设计]（🔴必做）可能导致：
- 模块划分不清 → 后期重构（概率：高，返工成本 3x）
- 目录结构混乱 → 维护困难（概率：中）

确认跳过？这将记录在交付清单的"遗留问题"中。(y/n)
```

### 场景 B：增量功能
> "加个搜索功能"

自动走增量精简流程：
1. 功能需求（1-2句）→ 2. 接口/DB 变更 → 3. 编码+测试 → 4. 验收

### 场景 C：直接跳编码
> "直接写代码吧"

```
理解。在开始前确认 3 个前提：
1. 需求和验收标准已明确？
2. 技术栈已确定？
3. 接口和数据模型已有方案？

都有 → 输出 1 页设计概要确认后立即编码
没有 → 前面花 30min 对齐，后面省 3 天返工。你怎么选？
```

```
📋调研→📝PRD→🎨原型→🔧选型→🏗️架构→🗄️DB→📡API→💻编码→🔍审查→🧪测试→🚀部署→✅交付
  ↓     ↓     ↓     ↓     ↓     ↓    ↓     ↓     ↓     ↓     ↓     🎉

### 标准模式（中型 · 8步）
```
📋调研→📝PRD→🎨原型→🔧选型+架构→🗄️DB→📡API→💻编码→✅测试交付
  ↓     ↓     ↓      ↓       ↓    ↓     ↓     🎉
```

### 快速模式（小型 · 4步）
```
📋需求确认→🗄️数据+接口→💻编码→✅测试验收
    ↓         ↓       ↓     🎉
```

每个 `↓` = 停下来等用户确认。🔴必做不可跳，🟡可精简不可跳。
